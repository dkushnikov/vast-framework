# VAST for Executives and Organizational Leaders

## What changes for you

VAST adds one layer to how you think about your organization: **Architecture** — between Vision and Strategy.

You already have Vision (where we're going) and Strategy (how we get there). Architecture says: **what is the system made of, and what constraints govern it?**

Without this layer, strategies are disconnected — teams optimize locally without a shared structural map. With it, every strategic decision references the same architecture.

## Concretely

**Before VAST:**
- Vision → Strategy → OKRs → Execution
- Architecture implicit — "everyone knows how things work"
- Values on the wall, not in decision-making
- OKRs sometimes degenerate into task lists

**With VAST:**
- Vision → Architecture → Strategy → Tactics
- Architecture explicit — domains named, constraints documented, interactions mapped
- Values anchored in Architecture as structural constraints (not aspirational posters)
- OKRs decomposed: Outcomes (Architecture) / Use Cases (Strategy) / Outputs (Tactics)

## One example

A team notices three data pipelines are broken. They fix them one by one. A month later, a fourth breaks.

With VAST: fixing three pipelines is Tactics. "No unified data protocol" is an Architecture gap. Naming the level changes what you build — instead of three fixes, you build the protocol that prevents the fourth.

This is VAST in one paragraph: **name the level, change the response.**

## What stays the same

- Vision is still yours to set
- OKRs still work — VAST gives them structural context, not a replacement
- Teams still own execution
- Existing org structure doesn't need to change — VAST names what's already there
- No new meetings, no new tooling, no reorg required

## What you need to decide

1. **Who owns Architecture?** Not necessarily CTO. Architecture = org design + decision frameworks + capability maps. Could be COO, could be cross-functional.
2. **Which areas need explicit Architecture?** Not everything. Start where implicit structure is causing disconnected strategies.
3. **How to introduce it?** Not as "a new framework" but as "let's make our implicit structure visible." Describe, don't prescribe.

## Common exec questions

**"Is this another framework?"** — VAST codifies what sophisticated leaders already do: think about structure before strategy. It names the step, doesn't invent it.

**"How is this different from OKRs?"** — OKRs are great for alignment. VAST adds structural context so OKRs connect to something. [See comparison →](../standard-framework.md)

**"What's the ROI?"** — Fewer rewrites from "architecture decisions made by default." Hard to quantify upfront; visible in first prevented rewrite.
